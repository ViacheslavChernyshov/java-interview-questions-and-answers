# Can you use multiple bounds for a single type parameter?

## Deep dive (Under the Hood)

Yes, Java supports multiple bounds for type parameters using the **`&`** symbol. This allows you to create highly specialized generics that must satisfy multiple interfaces or a class and interfaces simultaneously.

Syntax: `<T extends Class & Interface1 & Interface2>`.

---

## Senior insight: Rules and constraints (The Rules)

1.  **Class always comes first**: If there is a class among the bounds, it must be at the beginning of the list. Putting an interface before a class will result in a compilation error.
    - `T extends Number & Serializable` — ✅ Correct.
    - `T extends Serializable & Number` — ❌ Error.
2.  **Only one class**: Java forbids multiple class inheritance, so there can be at most one class in the bounds.
3.  **Erasure to first bound (Erasure to first bound)**: This is a critical nuance. After compilation, type `T` is replaced with the **first** type in the bounds list.
    - For `<T extends Number & Runnable>` the bytecode type will be `Number`.
    - Any access to `Runnable` methods will be accompanied by an implicit **`checkcast`** instruction in the bytecode.

---

## Senior insight: Implicit Intersection Types in Java 8+

With the introduction of lambdas and functional interfaces, intersection types started being used implicitly in the **Target Typing** mechanism:
```java
// A lambda that is both Runnable and Serializable
Object obj = (Runnable & Serializable) () -> System.out.println("Hello");

// This syntax is allowed ONLY in casts and type parameter declarations.
// You cannot declare a variable of type (Runnable & Serializable) x.
```
This is useful in distributed systems (Spark, Flink) where lambdas must be sent over the network.

---

## Why is it needed in architecture?

It allows avoiding the creation of "marker interfaces" that combine behavior.
- **Example**: A document processing method that requires an object to be `AutoCloseable` (for resource cleanup) AND `Comparable` (for sorting in a queue).
`public <T extends AutoCloseable & Comparable<T>> void process(T doc)`

---

## Performance and Highload

- **Bytecode bloat**: Additional bounds do NOT add extra runtime checks on every access. checkcast checks the first bound (erasure). Methods of the second/third bound require additional casts only when those specific methods are called.
- **Type Inference Latency**: Complex multiple bounds significantly slow down the compiler, as the type dependency graph becomes exponentially more complex.

---

## Edge Cases

- **Conflicting Signatures**: If two interfaces in the bounds have methods with the same name and parameters but different return types — such a generic cannot be implemented (compilation error).
- **Reflection**: The `getBounds()` method on a type will return an array of all bounds, allowing frameworks to do complex runtime type validation.

## Summary for Senior
- Multiple bounds — a tool for building **strict contracts**.
- Remember the **class priority** (first place in the list).
- The type erases to the **first bound** — choose it considering the most frequently used methods for optimization.
- Use intersection types in lambdas for serialization.

---

## 🎯 Interview Cheat Sheet

**Must know:**
- Multiple bounds: `<T extends Class & Interface1 & Interface2>` — via `&`
- Class always comes first: `T extends Number & Serializable`, not the other way around
- Only one class possible — Java single inheritance
- Type erasure to first bound: `T extends Number & Runnable` -> erasure = Number
- Intersection types in Java 8+: `(Runnable & Serializable) () -> {}` — for lambdas
- Conflicting signatures — two interfaces with same methods but different return types = error

**Frequent follow-up questions:**
- **Why must class come first?** — Erasure uses the first bound, single inheritance
- **Does bound order affect performance?** — Yes, first bound = erasure type, choose the most used one
- **Where is intersection type used?** — Lambdas for serialization: Spark, Flink require Serializable
- **What if interfaces have conflicting methods?** — Compilation error, cannot implement

**Red flags (DO NOT say):**
- ❌ "You can have multiple classes in bounds" — Java single inheritance, maximum one class
- ❌ "Bound order doesn't matter" — Class must be first, erasure to first bound
- ❌ "Multiple bounds give runtime overhead" — Only compile-time check, negligible
- ❌ "Intersection type can be used for variables" — Only in casts and type parameters
