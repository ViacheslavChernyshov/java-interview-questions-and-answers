# What is the difference between List<?> and List<Object>?

## 🟢 Junior Level

**`List<Object>`** — a concrete type: a list of objects.
- You can add any objects (`String`, `Integer`, etc.).
- **Invariant**: `List<Object>` is NOT a parent of `List<String>`.

```java
List<Object> objects = new ArrayList<>();
objects.add("Hello");
objects.add(42);  // ✅ allowed

// But:
List<String> strings = new ArrayList<>();
// process(objects);  // ✅
// process(strings);  // ❌ List<String> != List<Object>
```

**`List<?>`** — a list of unknown type.
- You can read elements only as `Object`.
- **Cannot write** (except `null`).
- **Covariant**: accepts `List<String>`, `List<Integer>`, etc.

```java
List<?> unknown = List.of("a", "b", "c");
Object o = unknown.get(0);  // ✅
// unknown.add("x");  // ❌ cannot write

List<String> strings = new ArrayList<>();
process(strings);  // ✅ List<?> accepts any parameterized List
```

---

## 🟡 Middle Level

### What happens at runtime

After type erasure, both types become just `List`. The difference is only in compile-time checks.

```java
// List<Object> — invariant
void process1(List<Object> list) { }
process1(new ArrayList<String>());  // ❌ compilation error

// List<?> — covariant
void process2(List<?> list) { }
process2(new ArrayList<String>());  // ✅ OK
```

### Why List<?> forbids writing

```java
List<?> unknown = new ArrayList<String>();
// unknown.add(42);  // ❌
```
The compiler doesn't know the actual element type. If writing `Integer` were allowed,
but the list is actually a `List<String>` — there would be a `ClassCastException` on read.

### Raw List vs List<?>

```java
// ❌ Raw type — all checks disabled
List raw = new ArrayList();
raw.add("Hello");
raw.add(42);  // compiler is silent!

// ✅ Wildcard — checks in place
List<?> wildcard = new ArrayList<String>();
wildcard.add(42);  // ❌ compilation error
```

---

## 🔴 Senior Level

### Reification and erasure

At the bytecode level, both options are simply `List`. `List<?>` is safer because
the compiler blocks unsafe operations, whereas raw type simply disables checks.

### List<?> does not give performance advantages

`List<?>` is not read-only by design. The compiler blocks writes because
it cannot verify type safety, not because it treats the collection as immutable.
No performance optimizations follow from this.

### Capture of Wildcard

```java
// Cannot do this:
void swap(List<?> list, int i, int j) {
    // var temp = list.get(i);
    // list.set(i, list.get(j));  // ❌ capture of ?
}

// Solution — capture helper:
void swap(List<?> list, int i, int j) {
    swapHelper(list, i, j);
}
private <T> void swapHelper(List<T> list, int i, int j) {
    T temp = list.get(i);
    list.set(i, list.get(j));
    list.set(j, temp);
}
```

### Best Practices

```java
// ✅ List<?> for arguments — maximum flexibility
void printAll(List<?> list) { }

// ✅ List<Object> only for heterogeneous containers
List<Object> mixed = new ArrayList<>();

// ❌ Raw List — disables all checks
// ❌ List<Object> when wildcard is needed
```

## 🎯 Interview Cheat Sheet

**Must know:**
- `List<Object>` — invariant, concrete type, can add any objects
- `List<?>` — covariant, unknown type, read only as Object, cannot write
- `List<String>` is NOT a subtype of `List<Object>` — generics are invariant
- `List<?>` accepts any parameterized List: `List<String>`, `List<Integer>`
- `List<?>` != raw `List` — wildcard preserves checks, raw disables all
- Capture of wildcard — cannot get/set for `?` in one method, need helper `<T>`

**Frequent follow-up questions:**
- **Can you add null to List<?>?** — Yes, null is valid for any type
- **How does List<?> differ from List<Object>?** — List<?> is covariant, List<Object> is invariant
- **What is capture of wildcard?** — The compiler cannot match the `?` type for writing
- **When to use List<Object>?** — Heterogeneous containers with different types

**Red flags (DO NOT say):**
- ❌ "List<?> is a read-only list" — The compiler blocks writes due to unknown type, not immutable
- ❌ "List<Object> accepts List<String>" — Generics are invariant, List<String> != List<Object>
- ❌ "List<?> gives performance advantages" — No runtime optimizations
- ❌ "List<?> and raw List are the same" — Wildcard preserves checks, raw disables them
