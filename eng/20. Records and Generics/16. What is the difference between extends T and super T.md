# What is the difference between <? extends T> and <? super T>?

## 🟢 Junior Level

**`<? extends T>`** — covariance (Producer).
- You can **read** elements as type `T`.
- **Cannot write** (except `null`).

```java
List<? extends Number> numbers = List.of(1, 2, 3);
Number n = numbers.get(0);  // ✅ can read
// numbers.add(4);  // ❌ cannot write
```

**`<? super T>`** — contravariance (Consumer).
- You can **write** elements of type `T`.
- Can only read as `Object`.

```java
List<? super Integer> integers = new ArrayList<>();
integers.add(42);  // ✅ can write
// Integer i = integers.get(0);  // ❌ can only read as Object
```

**Simple rule (PECS):**
- **P**roducer **E**xtends — reading data → `? extends T`
- **C**onsumer **S**uper — writing data → `? super T`

## 🟡 Middle Level

### Covariance — `<? extends T>`

Upper bound: type `T` or any of its subtypes.

```java
List<? extends Number> nums = new ArrayList<Integer>();
Number n = nums.get(0);  // ✅ Integer is a Number
// nums.add(42);  // ❌ compiler doesn't know if it's Integer or Double
```

**Why you cannot write:** `List<? extends Number>` could be a `List<Integer>` or `List<Double>`. If writing `Double` were allowed but the actual list is `List<Integer>` — integrity is broken.

### Contravariance — `<? super T>`

Lower bound: type `T` or any of its supertypes.

```java
List<? super Integer> target = new ArrayList<Number>();
target.add(42);  // ✅ Integer fits into Number or Object
Object o = target.get(0);  // ✅ can only read as Object
```

**Why you cannot read typed values:** `List<? super Integer>` could be `List<Number>` or `List<Object>`. There is no guarantee that `Integer` elements are stored there.

### Comparison

| Characteristic | `<? extends T>`          | `<? super T>`            |
| -------------- | ----------------------- | ----------------------- |
| **Role**       | Producer (source)       | Consumer (sink)         |
| **Read**       | ✅ As `T`                | ❌ Only as `Object`      |
| **Write**      | ❌ Forbidden             | ✅ As `T`                |
| **Hierarchy**  | `T` and all **below**   | `T` and all **above**   |

### Capture of Wildcard

Common mistake:
```java
void reverse(List<?> list) {
    // list.set(0, list.get(0)); // ERROR: capture of ?
}
```
The compiler cannot match the type you "got" with the type for writing.
**Solution:** A helper generic method that "captures" the type:
```java
void reverse(List<?> list) { reverseHelper(list); }
private <T> void reverseHelper(List<T> list) {
    // now T is known
}
```

## 🔴 Senior Level

### Use-site variance

Java uses variance **at the use site** — in method parameters.
This differs from Kotlin/Scala, which have **declaration-site variance** (`out T` / `in T`).

### Wildcards do NOT prevent allocations

Using wildcards in method signatures **does not prevent allocations**.
`ArrayList<Integer>` allocates `Integer` objects regardless of whether
it is passed as `List<? extends Number>`.
Wildcards are only compile-time type checking.

### Unbounded Wildcard `<?>`

This is effectively `<? extends Object>`. Useful when the code only uses `Object` methods:
```java
void printSize(List<?> list) {
    System.out.println(list.size());  // only Object methods
}
```

### Best Practices

```java
// ✅ extends — for extracting data (Producer)
public double sum(List<? extends Number> numbers) { ... }

// ✅ super — for accumulating data (Consumer)
public void fill(List<? super Integer> list, int value) { ... }

// ✅ No wildcard when you need both read and write
public <T> void swap(List<T> list, int i, int j) { ... }

// ❌ Extends for writing
// ❌ Super for reading a specific type
```

## 🎯 Interview Cheat Sheet

**Must know:**
- `<? extends T>` — covariance (Producer): can read as T, cannot write (except null)
- `<? super T>` — contravariance (Consumer): can write T, can only read as Object
- `List<String>` does NOT inherit from `List<Object>` — generics are invariant
- PECS rule: **P**roducer **E**xtends, **C**onsumer **S**uper
- Wildcards — use-site variance (unlike Kotlin's declaration-site variance)
- `<?>` = `<? extends Object>` — for cases when only Object methods are needed

**Frequent follow-up questions:**
- **Why does <? extends T> forbid writing?** — The compiler doesn't know the actual type (could be List<Integer>), writing would break integrity
- **What can you write into List<? super Integer>?** — Integer and its subclasses, because they fit into any supertype
- **What is capture of wildcard?** — The compiler cannot link get() with set() for `?`, a generic helper method is needed
- **When is wildcard NOT needed?** — When you need both read and write — use `<T>`

**Red flags (DO NOT say):**
- ❌ "List<? extends Number> could be List<Integer>" — Yes, but you cannot add Integer to it
- ❌ "<? super T allows reading as T" — You can only read as Object
- ❌ "Wildcards prevent allocations" — This is only a compile-time check
- ❌ "List<?> is a read-only list" — The compiler blocks writes due to unknown type, not immutable
