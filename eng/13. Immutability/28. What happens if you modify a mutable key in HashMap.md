# What happens if you modify a mutable key in HashMap?

---

## Junior Level

If you modify an object used as a key in `HashMap`, it gets **lost** ‚Äî you won't be able to find it.

```java
List<String> key = new ArrayList<>(List.of("A"));
Map<List<String>, String> map = new HashMap<>();
map.put(key, "value");

key.add("B"); // Modified the key!
System.out.println(map.get(key)); // null ‚Äî not found!
```

### Why

- On `put`, HashMap computed the hash of `["A"]` ‚Üí placed in bucket #5
- After key modification, hash of `["A", "B"]` ‚Üí different bucket
- `get` searches in the wrong bucket ‚Üí `null`

The object is still in the map, but you can't retrieve it.

---

## Middle Level

### Failure mechanics

1. **put(key, value)** ‚Äî key hash ‚Üí bucket #5
2. **Mutation** ‚Äî changed a key field ‚Üí new hash
3. **get(key)** ‚Äî new hash ‚Üí bucket #10 ‚Üí empty ‚Üí `null`

The object is physically still in bucket #5, but became **unreachable**.

### Consequences

**Logical unreachability**: the object cannot be retrieved via get() or removed via
remove() using the modified key. It occupies space until the entire map is cleared
or entrySet() is iterated.

**Duplicates in HashSet**:
```java
Set<List<String>> set = new HashSet<>();
set.add(new ArrayList<>(List.of("A")));
// Modified an element inside the list
set.add(new ArrayList<>(List.of("A"))); // Added again!
```

### How to find the lost object

Through full iteration ‚Äî **O(n)** instead of **O(1)**:
```java
for (var entry : map.entrySet()) {
    if (entry.getKey().contains("A")) {
        // found it
    }
}
```

---

## Senior Level

### How to avoid

1. **Immutable keys** ‚Äî `String`, `UUID`, `Record` ‚Äî always
2. **Remove ‚Üí modify ‚Üí add**:
   ```java
   V value = map.remove(key);
   key.mutate();
   map.put(key, value);
   ```
3. **IdentityHashMap** ‚Äî if mutation is inevitable and you don't control the object (comparison by `==`)
   ```java
   Map<List<String>, String> map = new IdentityHashMap<>();
   // Comparison by ==, not equals ‚Äî mutation doesn't break lookup as long as it's the same object.
   ```

### The deep problem

Modifying a mutable key breaks the HashMap **invariant**: the key's `hashCode` must be stable. This is not a HashMap bug, but a violation of the `hashCode()` contract ‚Äî if `equals` hasn't changed, then `hashCode` should be the same, but mutation changes both.

### Summary for Senior
- Modifying a mutable key "breaks" the map ‚Äî object is unreachable via `get()`
- This is a classic memory leak and source of hard-to-find bugs
- Always design keys as immutable entities
- If mutation is necessary: `remove` ‚Üí `mutate` ‚Üí `put`
- `IdentityHashMap` ‚Äî a last resort when key control is impossible

---

## Interview Cheat Sheet

**Must know:**
- Mutable key modification ‚Üí new hashCode ‚Üí get searches in a different bucket ‚Üí null
- Object is physically in the map, but became **unreachable** ‚Äî logical memory leak
- Duplicates in HashSet: modified object is added again as "new"
- Lost object can only be found via entrySet iteration ‚Äî O(n) instead of O(1)
- How to avoid: immutable keys always, or remove ‚Üí mutate ‚Üí put
- IdentityHashMap ‚Äî compares by `==`, mutation doesn't break lookup (as long as it's the same object)

**Frequent follow-up questions:**
- *Is the object removed from the map?* ‚Äî No, it stays in the old bucket, but is unreachable via get/remove
- *How to find a lost object?* ‚Äî Iterate entrySet ‚Äî O(n)
- *Does IdentityHashMap solve the problem?* ‚Äî Yes, compares by reference, but it's a specific use case
- *Is this a HashMap bug?* ‚Äî No, it's a contract violation: hashCode must be stable

**Red flags (do NOT say):**
- "The object is removed from the map" ‚Äî no, it stays but is unreachable
- "HashMap fixes the hashCode itself" ‚Äî no, it's the developer's contract
- "Mutable key is fine" ‚Äî classic source of memory leaks and bugs
- "IdentityHashMap is a replacement for HashMap" ‚Äî it's a completely different structure, compares by reference
