# В чём преимущество Decorator перед наследованием?

## 🟢 Junior Level

**Decorator** — паттерн, который добавляет новую функциональность объекту, **оборачивая** его в другой объект.

**Проблема наследования:** Если нужно добавить разные комбинации функций, получается взрыв классов.

**Пример с наследованием:**
```java
// ❌ 2^n классов для n функций
class Coffee { }
class MilkCoffee extends Coffee { }
class SugarCoffee extends Coffee { }
class MilkSugarCoffee extends Coffee { }  // Уже 4 класса!
class WhippedCreamMilkSugarCoffee extends Coffee { }  // И дальше больше...
```

**Решение — Decorator:**
```java
// ✅ n классов для n функций
interface Coffee { double getCost(); }

class SimpleCoffee implements Coffee {
    public double getCost() { return 1.0; }
}

// Декораторы
class MilkDecorator implements Coffee {
    private final Coffee coffee;
    public MilkDecorator(Coffee coffee) { this.coffee = coffee; }
    public double getCost() { return coffee.getCost() + 0.5; }
}

class SugarDecorator implements Coffee {
    private final Coffee coffee;
    public SugarDecorator(Coffee coffee) { this.coffee = coffee; }
    public double getCost() { return coffee.getCost() + 0.2; }
}

// Любая комбинация!
Coffee order = new MilkDecorator(
                   new SugarDecorator(
                       new SimpleCoffee()
                   )
               );
System.out.println(order.getCost()); // 1.7
```

**Главное преимущество:** Можно комбинировать декораторы как угодно, не создавая новые классы.

## 🟡 Middle Level

### Сравнение: Наследование vs Decorator

| Критерий | Наследование | Decorator |
|----------|-------------|-----------|
| **Когда добавляем** | Compile-time | Runtime |
| **Количество классов** | 2^n | n + 1 |
| **Комбинирование** | Фиксированное | Любое |
| **Гибкость** | Низкая | Высокая |

### Java I/O — классический пример

```java
// Decorator в действии
InputStream input = new BufferedInputStream(
                        new CipherInputStream(
                            new GZIPInputStream(
                                new FileInputStream("file.txt")
                            )
                        )
                    );

// Каждый слой — декоратор:
// FileInputStream → читает файл
// GZIPInputStream → разжимает данные
// CipherInputStream → расшифровывает
// BufferedInputStream → буферизует
```

### Когда НЕ использовать Decorator

```java
// ❌ Overengineering для простой функциональности
class LoggingUserService extends UserService {
    @Override
    public void create(User user) {
        log.info("Creating user: " + user);
        super.create(user);
    }
}

// ✅ Достаточно композиции или AOP
```

## 🔴 Senior Level

### Composition over Inheritance

```
Decorator = частный случай композиции

Наследование: "is-a" (Coffee IS-A Beverage)
Композиция: "has-a" (Decorator HAS-A Coffee)

→ Композиция гибче, менее хрупкая
```

### Dynamic Responsibility

```java
// Добавление/удаление обязанностей на runtime
public class Order {
    private List<OrderDecorator> decorators = new ArrayList<>();
    private BigDecimal basePrice;
    
    public void addDecorator(OrderDecorator d) { decorators.add(d); }
    public void removeDecorator(Class<? extends OrderDecorator> type) {
        decorators.removeIf(d -> d.getClass().equals(type));
    }
    
    public BigDecimal getPrice() {
        BigDecimal price = basePrice;
        for (OrderDecorator d : decorators) {
            price = d.calculate(price);
        }
        return price;
    }
}
```

### Production Experience

**Реальный сценарий: Java I/O Decorator**
- Spring: ResponseBodyAdvice = Decorator для HTTP ответов
- Добавляет логирование, сжатие, шифрование
- Без Decorator: пришлось бы наследовать контроллеры

### Best Practices

1. **Decorator** когда нужно runtime комбинирование
2. **Наследование** для "is-a" отношений
3. **Композиция** предпочтительнее наследования
4. **Интерфейс** у декоратора и компонента одинаковый
5. **AOP** может заменить простые декораторы

### Резюме для Senior

- **Decorator** = runtime добавление обязанностей
- **Композиция > Наследование** — гибче и менее хрупко
- **Java I/O** = классический пример Decorator
- **2^n vs n** — разница в масштабируемости
- **AOP** заменяет простые декораторы (@Transactional)
