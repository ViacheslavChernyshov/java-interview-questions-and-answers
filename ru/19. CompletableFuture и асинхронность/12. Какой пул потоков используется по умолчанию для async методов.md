# Какой пул потоков используется по умолчанию для async методов?

## Глубокое погружение (Under the Hood)

По умолчанию все асинхронные методы `CompletableFuture` (без параметра `Executor`) используют статический инстанс **`ForkJoinPool.commonPool()`**.

### Характеристики commonPool:
1.  **Параллелизм**: По умолчанию равен `Runtime.getRuntime().availableProcessors() - 1`.
2.  **Алгоритм**: Work-Stealing (воркеры "крадут" задачи из очередей друг друга).
3.  **Тип потоков**: Daemon-потоки. Они не мешают завершению работы JVM.
4.  **Class Loader**: Использует System Class Loader, что важно при работе в контейнерах серверов приложений.

---

## Senior-инсайт: Почему "Default" — это часто ловушка?

### 1. Проблема "Общей кухни"
`commonPool` используется **всей JVM**. 
- Parallel Streams, `CompletableFuture`, некоторые библиотеки (напр. Selenium или внутренние механизмы Spring) — все они бьются за одни и те же потоки.
- Если одна часть приложения запустит тяжелый стрим, асинхронные ответы в другом модуле начнут тормозить.

### 2. Блокировки (Starvation)
`commonPool` спроектирован для **CPU-bound** задач. 
- Если вы сделаете блокирующий I/O запрос (напр. `RestTemplate.get`) внутри `supplyAsync()`, вы займете поток из общего пула.
- Когда все потоки (обычно их 7 на 8-ядерном CPU) будут заняты ожиданием сети, приложение перестанет выполнять любые асинхронные задачи. Это называется **Thread Starvation**.

---

## Исключение: Малоядерные системы

Если на сервере только 1 ядро CPU, `CompletableFuture` не будет использовать `commonPool`.
- **Under the hood**: В этом случае CF будет создавать **новый поток на каждую задачу** (Thread-per-task). Это сделано для предотвращения дедлоков, но может "убить" сервер по памяти при большом количестве задач.

---

## Производительность и Highload

- **LIFO для локальных задач**: ForkJoin воркеры обрабатывают свои задачи в порядке LIFO. Это сохраняет "горячие" данные в кэше процессора (L1/L2), что критично для производительности.
- **Stealing в порядке FIFO**: Чтобы минимизировать конкуренцию, "крадущие" потоки забирают задачи из хвоста очереди.

---

## Когда использовать свой Executor? (Senior Rule)

1.  **I/O операции**: Всегда. Пул должен быть большим (напр. 50-100 потоков) или использовать Virtual Threads.
2.  **Изоляция**: Когда нужно гарантировать, что сбой в модуле "Уведомления" не завалит модуль "Оплата".
3.  **Мониторинг**: `commonPool` сложно мониторить. Кастомный `ThreadPoolExecutor` позволяет видеть размер очереди и количество отклоненных задач.

---

## Диагностика

- **`-Djava.util.concurrent.ForkJoinPool.common.parallelism=N`**: Системный флаг для изменения размера пула.
- **`ForkJoinPool.getCommonPoolParallelism()`**: Программный способ узнать размер.
- **`jstack`**: Потоки называются `ForkJoinPool.commonPool-worker-X`. Если вы видите их в состоянии `WAITING` на сетевых сокетах — у вас архитектурная проблема.

## Резюме для Senior
- Пул по умолчанию — **ForkJoinPool.commonPool()**.
- Он только для **быстрых вычислений**.
- Блокировки в `commonPool` — это преступление против стабильности системы.
- В серьезных проектах всегда внедряйте свои `Executors` через Spring `@Bean`.
