# Как создать CompletableFuture, который уже завершён с результатом?

## Глубокое погружение (Under the Hood)

Создание уже завершенного **CompletableFuture (CF)** — это не просто удобный способ инициализации, это важный инструмент оптимизации и соблюдения контрактов в асинхронном API.

## Основные методы создания

### 1. CompletableFuture.completedFuture(value)
Создает CF, у которого поле `result` уже заполнено значением, а стек коллбэков пуст.
- **Under the hood**: При вызове `get()` или `join()` результат будет возвращен мгновенно без переключения контекста потока.

### 2. CompletableFuture.failedFuture(Throwable) (Java 9+)
Создает CF, завершенный с ошибкой.
- **Senior Insight**: До Java 9 приходилось делать `new CompletableFuture()` и сразу звать `completeExceptionally()`. Новый метод чище и эффективнее.

---

## Senior-инсайт: Зачем это нужно в реальных проектах?

### 1. Условная асинхронность (Caching)
Если данные есть в кэше, нам не нужно занимать поток в `ExecutorService`. Мы возвращаем завершенный CF.
```java
public CompletableFuture<Data> getData(String id) {
    Data cached = cache.get(id);
    if (cached != null) {
        return CompletableFuture.completedFuture(cached); // Мгновенный ответ
    }
    return CompletableFuture.supplyAsync(() -> db.fetch(id)); // Асинхронный запрос
}
```

### 2. Реализация паттерна "Null Object" или Fallback
Если мы хотим вернуть пустой результат, который не сломает цепочку обработки.
`return CompletableFuture.completedFuture(Collections.emptyList());`

### 3. Мокирование и тестирование
В Unit-тестах асинхронных сервисов использование `completedFuture` позволяет тестировать логику цепочек (`thenApply`, `thenCompose`) в синхронном режиме, что делает тесты быстрыми и детерминированными.

---

## Побочный эффект: Поток выполнения (Threading)

Это критический нюанс для Senior-разработчика.
Если вы вызываете `.thenApply()` на **уже завершенном** CF:
- **Коллбэк выполнится в ТЕКУЩЕМ потоке** (в том, который вызывает `thenApply`).
- Если CF еще **не завершен**, коллбэк выполнится в потоке, который завершит CF (обычно воркер из `ForkJoinPool`).
- **Следствие**: Если вы ожидали асинхронности, но получили завершенный CF, ваш "тяжелый" коллбэк может заблокировать главный поток (UI или HTTP-воркер). 
  - **Решение**: Всегда используйте `*Async` версии методов, если не уверены в источнике CF.

---

## Производительность и Highload

- **Allocation**: `completedFuture` создает объект в куче. В экстремальных случаях (миллионы запросов в сек) это может нагружать GC.
- **Project Valhalla**: В будущем CF может стать Value-типом (или быть оптимизирован через Scalar Replacement), что уберет оверхед на аллокацию.

---

## Пограничные случаи (Edge Cases)

- **Failed Future vs exceptionally**: Помните, что `failedFuture` сразу переводит цепочку в состояние ошибки. Если в цепочке есть `thenApply`, они будут пропущены.
- **CompletableFuture.completedStage(value)** (Java 12+): Возвращает интерфейс `CompletionStage`, который нельзя завершить вручную (иммутабельное представление). Это безопаснее для публичных API.

## Резюме для Senior
- Используйте `completedFuture` для оптимизации путей с кэшем.
- Помните, что синхронные методы на завершенном CF работают в текущем потоке.
- Для возврата ошибки используйте `failedFuture` (Java 9+).
- Это основа для построения гибких и быстрых асинхронных сервисов.
