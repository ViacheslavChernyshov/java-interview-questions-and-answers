# Как указать свой Executor для CompletableFuture?

## Глубокое погружение (Under the Hood)

Передача собственного `Executor` в методы `CompletableFuture` — это единственный способ обеспечить **изоляцию ресурсов** и **предсказуемость** системы под нагрузкой. Все методы с суффиксом `*Async` имеют перегруженную версию, принимающую `Executor`.

---

## Senior-инсайт: Стратегии выбора пула

На уровне Senior+ вы должны разделять потоки по типу нагрузки:

### 1. I/O-Bound (HTTP, БД, Файлы)
Используйте `ThreadPoolExecutor` с большим количеством потоков и ограниченной очередью.
```java
Executor ioExecutor = new ThreadPoolExecutor(
    10, 100, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000), // Ограничение очереди обязательно!
    new ThreadFactoryBuilder().setNameFormat("io-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // Backpressure
);
```

### 2. CPU-Bound (Расчеты, Маппинг)
Используйте `FixedThreadPool` размером в количество ядер или `commonPool`.

### 3. Virtual Threads (Java 21+)
Идеальный выбор для `CompletableFuture`, заменяющий огромные I/O пулы.
```java
Executor vtExecutor = Executors.newVirtualThreadPerTaskExecutor();
CompletableFuture.supplyAsync(task, vtExecutor);
```

---

## Senior-инсайт: Ловушка передачи Executor

Важно помнить, что `Executor` нужно передавать на **каждом** асинхронном этапе цепочки, если вы хотите сохранить контекст выполнения.

```java
// Плохо: только первый этап в своем пуле, остальные - в commonPool
CF.supplyAsync(task, myExecutor)
  .thenApplyAsync(transform); // Использует commonPool!

// Хорошо: изоляция сохраняется
CF.supplyAsync(task, myExecutor)
  .thenApplyAsync(transform, myExecutor);
```

## Продвинутые техники: ManagedBlocker

Если вы используете `ForkJoinPool` (в т.ч. `commonPool`) для задач, которые могут блокироваться, вы можете использовать интерфейс **`ManagedBlocker`**.
- **Under the hood**: Это позволяет пулу динамически создавать новый поток, чтобы компенсировать заблокированный, сохраняя уровень параллелизма. Это сложнее в реализации, чем просто `FixedThreadPool`, но гораздо эффективнее.

---

## Производительность и Мониторинг

- **Thread Naming**: Всегда используйте `ThreadFactory` (напр. из Guava или Spring), чтобы давать потокам имена. Без этого анализ дампов памяти в случае инцидента будет невозможен.
- **Reject Policy**: При переполнении пула `CompletableFuture` может бросить `RejectedExecutionException`. Используйте `CallerRunsPolicy`, чтобы замедлить продюсера запросов (естественный Backpressure).

---

## Пограничные случаи (Edge Cases)

- **Lifecycle**: Не забывайте вызывать `shutdown()` для ваших кастомных экзекуторов при выключении приложения (через `@PreDestroy` в Spring). Иначе вы получите "зомби-потоки", мешающие корректному завершению процесса.
- **Spring @Async**: Если вы используете `CompletableFuture` совместно с `@Async`, Spring может подставить свой `TaskExecutor`. Убедитесь, что вы контролируете его конфигурацию.

## Резюме для Senior
- Передача `Executor` — это стандарт для Production.
- Разделяйте I/O и CPU нагрузки в разные пулы.
- Используйте `CallerRunsPolicy` для защиты от перегрузки.
- На Java 21 переходите на `VirtualThreadPerTaskExecutor` для I/O задач.
