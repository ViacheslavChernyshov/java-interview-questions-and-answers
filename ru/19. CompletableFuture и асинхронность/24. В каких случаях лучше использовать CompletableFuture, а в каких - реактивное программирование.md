# В каких случаях лучше использовать CompletableFuture, а в каких — реактивное программирование?

## Глубокое погружение (Under the Hood)

Выбор между **CompletableFuture (CF)** и реактивными библиотеками (напр. **Project Reactor / WebFlux**) — это выбор между "одиночным выстрелом" и "потоком данных".

### CompletableFuture (Single-shot)
- **Модель**: Паттерн **Promise**. Одно вычисление -> Один результат (или ошибка).
- **Контекст**: Идеально для 90% задач в микросервисах: сходить в БД, вызвать API, объединить результаты.
- **Оверхед**: Минимальный. Часть стандартной библиотеки JDK.

### Reactive Programming (Streams)
- **Модель**: Паттерн **Observer + Iterator + Backpressure**. Поток событий (0..N элементов).
- **Контекст**: High-throughput системы, работа с WebSocket, стриминг данных, обработка бесконечных логов.
- **Оверхед**: Заметный (сложные стейт-машины внутри операторов). Требует сторонних зависимостей.

## Senior-инсайт: Когда CF становится недостаточно?

### 1. Проблема Backpressure
В `CompletableFuture` нет механизма обратного давления. Если продюсер генерирует данные быстрее, чем консьюмер их обрабатывает через `thenAccept`, очередь в `Executor` будет расти до бесконечности, пока не случится `OutOfMemoryError`. 
- **Реактивный подход**: Консьюмер сам запрашивает количество элементов, которое он может переварить (request-n).

### 2. Потоковая обработка
Если вам нужно обрабатывать данные по мере их поступления (напр. чанк за чанком из S3), CF заставит вас либо собрать всё в память, либо городить сложные рекурсивные цепочки. Реактивный `Flux` делает это "из коробки" одной строчкой кода.

## Сравнение в контексте Highload

| Характеристика | CompletableFuture | Reactive (Reactor/RxJava) |
| :--- | :--- | :--- |
| **Количество результатов** | Ровно 1 | 0 ... ∞ |
| **Backpressure** | ❌ Нет | ✅ Да |
| **Операторы** | Базовые (~20) | Богатые (~200: zip, flatMap, switchIfEmpty, retry...) |
| **Сложность отладки** | Средняя | Экстремально высокая (потеря стек-трейсов) |
| **Кривая обучения** | Низкая | Очень высокая |

## Senior-инсайт: Эффект Virtual Threads (Java 21)

С появлением **Virtual Threads**, необходимость в реактивном программировании и даже в сложных цепочках `CompletableFuture` для типичных I/O задач **снижается**.
- Теперь можно писать простой синхронный код, который масштабируется так же хорошо, как реактивный.
- Реактивность остается актуальной там, где важна именно **событийная природа** (Streaming) и **Backpressure**.

---

## Рекомендации по выбору (Decision Matrix)

1.  **CompletableFuture**: Если нужно сделать 2-3 параллельных вызова API и объединить их.
2.  **CompletableFuture + Virtual Threads**: Для масштабируемых блокирующих операций.
3.  **Reactive**: Если вы строите шлюз для обработки миллионов мелких событий, работаете с Kafka Streams или вам нужен полный контроль над нагрузкой (Backpressure).

## Резюме для Senior
- CF — для асинхронных **запросов**.
- Реактивность — для асинхронных **потоков**.
- Не используйте реактивность там, где достаточно CF; сложность поддержки реактивного кода огромна.
- Помните про **Project Valhalla** и **Loom** — они делают CF и синхронный код более эффективными.
