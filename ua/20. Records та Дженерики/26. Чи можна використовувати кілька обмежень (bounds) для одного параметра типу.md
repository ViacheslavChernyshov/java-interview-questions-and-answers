# Чи можна використовувати кілька обмежень (bounds) для одного параметра типу

## Глибоке занурення (Under the Hood)

Так, Java підтримує множинні обмеження для параметрів типу за допомогою символу **`&`**. Це дозволяє створювати високоспеціалізовані дженерики, які повинні задовольняти одразу кільком інтерфейсам або класу та інтерфейсам.

Синтаксис: `<T extends Class & Interface1 & Interface2>`.

---

## Senior-інсайт: Правила та обмеження (The Rules)

1.  **Клас завжди перший**: Якщо серед обмежень є клас, він зобов'язаний стояти на початку списку. Спроба поставити інтерфейс перед класом призведе до помилки компіляції.
    - `T extends Number & Serializable` — ✅ Правильно.
    - `T extends Serializable & Number` — ❌ Помилка.
2.  **Тільки один клас**: В Java заборонено множинне наслідування класів, тому і в обмеженнях може бути максимум один клас.
3.  **Стирання до першої границі (Erasure to first bound)**: Це критичний нюанс. Після компіляції тип `T` замінюється на **перший** тип в списку обмежень.
    - Для `<T extends Number & Runnable>` типом в байт-коді буде `Number`.
    - Будь-яке звернення до методів `Runnable` супроводжуватиметься неявною інструкцією **`checkcast`** в байт-коді.

---

## Senior-інсайт: Неявні Intersection Types в Java 8+

З введенням лямбд та функціональних інтерфейсів, перетин типів став використовуватися неявно в механізмі **Target Typing**:
```java
// Лямбда, яка одночасно є Runnable і Serializable
Object obj = (Runnable & Serializable) () -> System.out.println("Hello");

// Цей синтаксис допустимий ТІЛЬКИ в casts та type parameter declarations.
// Не можна оголосити змінну типу (Runnable & Serializable) x.
```
Це корисно в розподілених системах (Spark, Flink), де лямбди повинні передаватися по мережі.

---

## Навіщо це потрібно в архітектурі?

Це дозволяє уникати створення "інтерфейсів-пустушок", що об'єднують поведінку.
- **Приклад**: Метод обробки документів, який вимагає, щоб об'єкт був `AutoCloseable` (для очищення ресурсів) І `Comparable` (для сортування в черзі).
`public <T extends AutoCloseable & Comparable<T>> void process(T doc)`

---

## Продуктивність та Highload

- **Bytecode bloat**: Додаткові bounds НЕ додають extra runtime перевірки при кожному доступі. checkcast перевіряє перший bound (erasure). Методи другого/третього bound вимагають додаткових cast тільки при виклику цих конкретних методів.
- **Type Inference Latency**: Складні множинні границі значно сповільнюють роботу компілятора, оскільки граф залежностей типів стає експоненціально складнішим.

---

## Пограничні випадки (Edge Cases)

- **Conflicting Signatures**: Якщо два інтерфейси в обмеженнях мають методи з однаковим іменем та параметрами, але різними повертаними типами — такий дженерик буде неможливо реалізувати (помилка компіляції).
- **Reflection**: Метод `getBounds()` у типу поверне масив всіх границь, що дозволяє фреймворкам робити складну валідацію типів в рантаймі.

## Резюме для Senior
- Множинні границі — інструмент побудови **строгих контрактів**.
- Пам'ятайте про **пріоритет класу** (перше місце в списку).
- Тип стирається до **першої границі** — обирайте її з урахуванням найбільш використовуваних методів для оптимізації.
- Використовуйте перетин типів в лямбдах для серіалізації.

## 🎯 Шпаргалка для співбесіди

**Обов'язково знати:**
- Multiple bounds: `<T extends Class & Interface1 & Interface2>` — через `&`
- Клас завжди перший в списку: `T extends Number & Serializable`, не навпаки
- Тільки один клас можливий — Java single inheritance
- Type erasure до першого bound: `T extends Number & Runnable` -> erasure = Number
- Intersection types в Java 8+: `(Runnable & Serializable) () -> {}` — для лямбд
- Conflicting signatures — два інтерфейси з однаковими методами але різними return types = помилка

**Часті уточнюючі запитання:**
- **Чому клас повинен бути першим?** — Erasure використовує перший bound, single inheritance
- **Чи впливає порядок bounds на продуктивність?** — Так, перший bound = erasure type, обирайте найбільш використовуваний
- **Де використовується intersection type?** — Лямбди для серіалізації: Spark, Flink вимагають Serializable
- **Що якщо інтерфейси мають конфліктуючі методи?** — Compilation error, неможливо реалізувати

**Червоні прапорці (НЕ говорити):**
- ❌ "Можна кілька класів в bounds" — Java single inheritance, максимум один клас
- ❌ "Порядок bounds не важливий" — Клас зобов'язаний бути першим, erasure до першого bound
- ❌ "Multiple bounds дають runtime overhead" — Тільки compile-time перевірка, negligible
- ❌ "Intersection type можна використовувати для змінних" — Тільки в casts та type parameters
