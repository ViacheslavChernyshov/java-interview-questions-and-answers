# В чому перевага Decorator перед наслідуванням?

## 🟢 Junior Level

**Decorator** — патерн, який додає нову функціональність об'єкту, **огортаючи** його в інший об'єкт.

**Проблема наслідування:** Якщо потрібно додати різні комбінації функцій, виходить вибух класів.

**Приклад з наслідуванням:**
```java
// ❌ 2^n класів для n функцій
class Coffee { }
class MilkCoffee extends Coffee { }
class SugarCoffee extends Coffee { }
class MilkSugarCoffee extends Coffee { }  // Вже 4 класи!
class WhippedCreamMilkSugarCoffee extends Coffee { }  // І далі більше...
```

**Рішення — Decorator:**
```java
// ✅ n класів для n функцій
interface Coffee { double getCost(); }

class SimpleCoffee implements Coffee {
    public double getCost() { return 1.0; }
}

// Декоратори
class MilkDecorator implements Coffee {
    private final Coffee coffee;
    public MilkDecorator(Coffee coffee) { this.coffee = coffee; }
    public double getCost() { return coffee.getCost() + 0.5; }
}

class SugarDecorator implements Coffee {
    private final Coffee coffee;
    public SugarDecorator(Coffee coffee) { this.coffee = coffee; }
    public double getCost() { return coffee.getCost() + 0.2; }
}

// Будь-яка комбінація!
Coffee order = new MilkDecorator(
                   new SugarDecorator(
                       new SimpleCoffee()
                   )
               );
System.out.println(order.getCost()); // 1.7
```

**Головна перевага:** Можна комбінувати декоратори як завгодно, не створюючи нові класи.

## 🟡 Middle Level

### Порівняння: Наслідування vs Decorator

| Критерій               | Наслідування  | Decorator                         |
| ---------------------- | ------------- | --------------------------------- |
| **Коли додаємо**       | Compile-time  | Runtime                           |
| **Кількість класів**   | 2^n           | n декораторів + базовий компонент |
| **Комбінування**       | Фіксоване     | Будь-яке                          |
| **Гнучкість**          | Низька        | Висока                            |

### Java I/O — класичний приклад

```java
// Decorator в дії
InputStream input = new BufferedInputStream(
                        new CipherInputStream(
                            new GZIPInputStream(
                                new FileInputStream("file.txt")
                            )
                        )
                    );

// Кожен шар — декоратор:
// FileInputStream → читає файл
// GZIPInputStream → стискає дані
// CipherInputStream → розшифровує
// BufferedInputStream → буферизує
```

### Коли НЕ використовувати Decorator

```java
// ❌ Overengineering для простої функціональності
class LoggingUserService extends UserService {
    @Override
    public void create(User user) {
        log.info("Creating user: " + user);
        super.create(user);
    }
}

// ✅ Достатньо композиції або AOP
```

## 🔴 Senior Level

### Composition over Inheritance

```
Decorator = окремий випадок композиції

Наслідування: "is-a" (Coffee IS-A Beverage)
Композиція: "has-a" (Decorator HAS-A Coffee)

→ Композиція гнучкіша, менш крихка
```

### Dynamic Responsibility

```java
// Це Chain of Responsibility / Pipeline, а не класичний Decorator.
// Класичний Decorator — вкладені обгортки: new Compression(new Encryption(new Stream()))
// Додавання/видалення обов'язків на runtime
public class Order {
    private List<OrderDecorator> decorators = new ArrayList<>();
    private BigDecimal basePrice;

    public void addDecorator(OrderDecorator d) { decorators.add(d); }
    public void removeDecorator(Class<? extends OrderDecorator> type) {
        decorators.removeIf(d -> d.getClass().equals(type));
    }

    public BigDecimal getPrice() {
        BigDecimal price = basePrice;
        for (OrderDecorator d : decorators) {
            price = d.calculate(price);
        }
        return price;
    }
}
```

### Production Experience

**Реальний сценарій: Java I/O Decorator**
- Spring: ResponseBodyAdvice = Decorator для HTTP відповідей
- Додає логування, стиснення, шифрування
- Без Decorator: довелося б наслідувати контролери

### Best Practices

1. **Decorator** коли потрібне runtime комбінування
2. **Наслідування** для "is-a" відносин
3. **Композиція** пріоритетніша за наслідування
4. **Інтерфейс** у декоратора і компонента однаковий
5. **AOP** може замінити прості декоратори

### Резюме для Senior

- **Decorator** = runtime додавання обов'язків
- **Композиція > Наслідування** — гнучкіше і менш крихке
- **Java I/O** = класичний приклад Decorator
- **2^n vs n** — різниця у масштабованості
- **AOP** замінює прості декоратори (@Transactional)

## 🎯 Шпаргалка для інтерв'ю

**Обов'язково знати:**
- Decorator додає функціональність через обгортку, наслідування — через розширення класу
- Масштабованість: Decorator = n класів для n функцій, наслідування = 2^n класів (комбінаторний вибух)
- Java I/O — класичний приклад: BufferedInputStream(CipherInputStream(GZIPInputStream(FileInputStream)))
- Composition over Inheritance: Decorator — окремий випадок композиції, менш крихкий ніж наслідування
- Decorator працює на runtime, наслідування — на compile-time
- AOP (@Transactional, @Cacheable) замінює прості декоратори
- Динамічне додавання/видалення декораторів на runtime (Pipeline/Chain патерн)

**Часті уточнювальні запитання:**
- **Коли Decorator — overengineering?** — Проста функціональність, достатньо композиції або AOP
- **Чим Decorator відрізняється від Adapter?** — Decorator додає функціональність, Adapter змінює інтерфейс
- **Чому Java I/O — приклад Decorator?** — Кожен stream огортає інший, додаючи свою функцію
- **Чи можна видалити декоратор на runtime?** — Так, якщо зберігати список декораторів (Pipeline патерн)

**Червоні прапорці (НЕ говорити):**
- "Наслідування завжди краще за Decorator" — комбінаторний вибух класів (2^n)
- "Decorator і Proxy — це одне й те саме" — Proxy контролює доступ, Decorator додає функціональність
- "Я використовую Decorator для логування" — AOP (@Aspect) підходить краще
- "Decorator складний для розуміння" — Java I/O використовує його, він інтуїтивний
